                                        event-object.js

- The callback function can get an event parameter
- without event parameter: const btnCBFunction = () => alert('You clicked me');
- with event parameter:
                        const btnCBFunction = (e) => {
                        console.log(e);
                        alert('You clicked me');
                        };

- with event parameter and another parameter
                        const cbFunction = (event, item) => {
                            console.log(e.target);
                            console.log(e.currentTarget);
                        }
                        // registering callback
                        skillsList.forEach((item) => {
                        item.addEventListener('click', (event) => cbFunction(event, item));
                        });

- event e is of type PointerEvent
-- event.type: 'click'
-- event.target: The element that has been clicked
-- event.currentTarget: The element that fires the event (for which the event/callback is registered)

Imagine this HTML:

<li class="item">
  <span>Click me</span>
</li>

And your JS attaches the listener to <li>.

If the user clicks on the <span>:

event.target → <span> (the real clicked element)
event.currentTarget → <li> (where the listener is attached)

--------------------------------------------------------------------------------------------

const windowCBFunction = (event, msg) => {
  console.log(event.currentTarget);
  console.log(msg);
};

skillsH2.addEventListener('click', (event) => h2CBFunction(event, 'You clicked H2 inside Skills section'));
window.addEventListener('click', (event) => windowCBFunction(event, 'You clicked Window'));

- Clicking the window any where will call windowCBFunction()
- Clicking the element skillsH2 will call h2CBFunction() and windowCBFunction()
-- Cliking the mouse will click the window first and then skillsH2
-- But why h2CBFunction() is fired first and then windowCBFunction()? -> Bubbling & Capture

- Bubbling:
عند حدوث حدث في عنصر فرعي (child element) يتم تنفيذ الحدث في العنصر الفرعي أولاً ثم ينتقل لتنفيذ الأحداث في العناصر الأعلى  (أي العنصر الأب والعنصر الجد وهكذا). هذا هو السلوك الافتراضي للأحداث في JavaScript.

- Capture:
يعمل بالاتجاه المعاكس لـ Bubbling. عند حدوث حدث في عنصر فرعي يتم تنفيذ الحدث في العنصر الأعلى أولاً ثم ينتقل لتنفيذ الحدث في العنصر الفرعي. وهو يتيح لك التحكم في تنفيذ الأحداث في العناصر الأعلى قبل تنفيذها في العناصر الفرعية.

- To prevent bubbling use event.stopPropagation(). In this case only h2CBFunction() is called. windowCBFunction() is NOT called
stopPropagation
هو طريقة تُستخدم لمنع انتشار (Propagation) الحدث (سواء كان Bubbling أو Capture) بعد تنفيذه في العنصر الحالي. عند استخدام هذه الطريقة لا يتم تنفيذ الأحدث في العناصر الأعلى أو الأدنى بعد تنفيذه في العنصر الحالي.
    const h2CBFunction = (event, msg) => {
        event.stopPropagation();
        console.log(event.currentTarget);
        console.log(msg);
    };

capture: true:
عند تعيين capture=true يتم تنفيذ الحدث باستخدام النمط capture بدلاً من النمط bubbling. يعني ذلك أن الحدث سينتقل من العنصر الأعلى إلى العنصر الأدنى. بمعنى آخر سيتم تنفيذ الحدث الموجود في العنصر الأعلى مثل الـ Window قبل أن يتم تنفيذه في العنصر الأدني مثل h2.

مع استخدام stopPropagation و capture: true يُمكنك التحكم بدقة في تنفيذ الأحداث.

window.addEventListener('click', (event) => windowCBFunction(event, 'You clicked Window'),
{   // 3rd parameter of addEventListener is optional
  capture: true,
});

- in this case windowCBFunction() is called first and then h2CBFunction()

role="button" is an ARIA (Accessible Rich Internet Applications) role you can add to an HTML element to tell assistive technologies (screen readers, etc.) that the element should be treated like a button. It does not by itself provide native button behavior (focus, keyboard activation, pressed state) — you must add those behaviors.

What it does

Announces the element to assistive tech as a "button".
Does not automatically give keyboard support, focusability, or disabled semantics.
When to use

Use role="button" only when you cannot use a native <button> (for example, a complex custom widget, SVG, or third‑party markup you cannot change).
Prefer native <button> whenever possible — it already supports accessibility, keyboard, focus, and form behaviors.
What you must add when using role="button"

Make it focusable: add tabindex="0" so it is reachable by Tab.
Implement keyboard activation:
Enter should activate immediately (keydown).
Space should activate on keyup (ARIA authoring practices) and often you should prevent default on keydown to stop the page from scrolling.
Provide an accessible name: visible text, aria-label, or aria-labelledby.
For toggle buttons, include aria-pressed="true|false".
For disabled state, use aria-disabled="true" and prevent activation.
